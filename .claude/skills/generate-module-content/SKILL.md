# Generate Module Content

## Overview

Starting from a curriculum that provides the list of modules of the language course, generate or update content for the requested module. "Content" means four artifacts, generated in order: the **vocabulary set**, the **grammar concepts**, the **exercise bank** and the **module** root file.

## When to Use

- The user requests the creation of content for a **given module** in the curriculum.
- The user requests changes in the generated content for a **given module**. 

**Trigger Phrases:**
- "Generate content for module A2-12"
- "Generate module A2-06"

## Prerequisites

- You have read `docs/specs/language-learning/idea.md` — **important** for understanding the context and core concepts.
- You have read `docs/specs/language-learning/data-model.md` — **important**: the generated files MUST match these schemas (`VocabularyItem`, `GrammarConcept`, `Exercise`).
- You have read `docs/specs/language-learning/default-modules.md` — **important** for an overview of the whole curriculum and to locate the target module's shell.
- You **MUST** have been told which module to generate content for (e.g. `A2-01`). The module **must** exist in `default-modules.md`.
- You **MUST** ask the user where to store generated files. The user must clearly provide **the path to a folder**. This is known going forward as `<user-specified-folder>`.

**Reference material and tools bundled with this skill:**
- `rules-for-generation.md` — the binding rules for exercise generation (Phase 3). Read before Phase 3.
- `module-examples.md` — worked examples of vocabulary sets and all six exercise types, across every CEFR level. Use it as the gold standard for tone, vocabulary sizing, and exercise style. Read before Phase 1.
- `generate_ids.py` — **must** be used to generate all `id` values. Never hand-write an id. See ID Rules below.

## The Workflow

You will follow 5 phases. Each phase ends with a **self-validation** step. Do not advance until the current phase passes its checklist. No human review is required (per idea.md §3.1.3) — you are responsible for validating your own output against the data model and rules. Surface a one-line summary to the user at the end of each phase.

```
GENERATE VOCABULARY ITEMS > GENERATE GRAMMAR CONCEPTS > GENERATE EXERCISES > COVERAGE & DISTRIBUTION QA > GENERATE MODULE
```

First, **locate the target module's shell** in `default-modules.md` (theme, communication goal, grammar focus, vocabulary focus). This shell is your specification for all three phases.

In general, if the user is asking for an **update** of the content or changes in the content, use this workflow but in "edit" mode: don't rewrite everything, understand the delta and update the parts that need to be corrected.

## ID Rules

### Module ID

The module's own `id` is **not** generated by the script. Derive it from the module code and the language:

```
danish-<cefrLevel>-<sequence>   →  danish-A2-02
```

- Language prefix is always `danish` for this curriculum.
- `<cefrLevel>` and `<sequence>` come directly from the module code in `default-modules.md` (e.g. `A2-02` → level `A2`, sequence `02`).

Use this id as:
- The `id` field in the Phase 5 module file.
- The `moduleId` field on **every exercise** in the Phase 3 exercise bank.

### Vocabulary and Grammar Concept IDs

All `id` values for vocabulary items and grammar concepts **must** be generated by running `generate_ids.py`. Never invent or hand-write an id.

**Format:**
```
<module>-v-<name>-<random>     →  A2-02-v-kaffe-7392
<module>-g-<name>-<random>     →  A2-02-g-inversion-4821
```

- `<name>` is short, dash-separated, max 4 words (the script enforces this and handles Danish characters).
- `<random>` is a 4-digit number generated by the script.

**How to run** (pass all names in one call using a JSON array — required when any name is a phrase):
```bash
# Vocabulary items for module A2-02 (words and phrases mixed)
python3 .claude/skills/generate-module-content/generate_ids.py A2-02 v \
  '["kaffe", "mælk", "Jeg vil gerne have...", "Kan jeg få?", "sulten"]'

# Grammar concepts for module A2-02
python3 .claude/skills/generate-module-content/generate_ids.py A2-02 g \
  '["inversion", "negation placement"]'
```

The script outputs a JSON array of `{"name": ..., "id": ...}` pairs. Use those ids verbatim in the generated files.

**Note on grammar concept ids:** Grammar concepts are canonical and shared across modules (e.g. "Inversion" appears in A2-14 and B1-02). The id generated here is module-scoped and serves as the stable reference *within the files you generate for this module*. At database import time, concepts are deduplicated by `name` — so two modules that both generate an "Inversion" concept will map to the same database record regardless of their local ids.

### Phase 1: Generate Vocabulary Items

- Generate **all vocabulary items needed** to genuinely cover the module's **vocabulary focus** and support its **communication goal**.
- The count **must** emerge from the content — don't pad, don't cut. Typical range is **15–40 items**; a numbers/calendar module sits at the high end, a meta-communication module at the low end.
- Every item's `type` **must** be drawn from the taxonomy in idea.md §3.2.1 (`noun`, `verb`, `adjective`, `adverb`, `phrase`, `pattern`, `connector`, `pronoun`, `number`).
- Set `context` only when the word needs disambiguation (e.g. `stor` → "physical size"); otherwise `null`.
- `source` is always `curriculum`.
- **Before writing the file**, run `generate_ids.py` for all vocabulary items in one call (see ID Rules). Use the returned ids verbatim.

**Output schema** (array of `VocabularyItem`, matching data-model.md):
```json
{
  "id": "A2-02-v-kaffe-7392",
  "danish": "kaffe",
  "english": "coffee",
  "type": "noun",
  "context": null,
  "tags": ["food", "drink"],
  "cefrLevel": "A1",
  "source": "curriculum"
}
```

**Expected file:** `<user-specified-folder>/<module>/<module>-vocabulary.json` (e.g. `A2-02`).

**Self-validation before advancing:**
- Every item covers part of the module's vocabulary focus / communication goal.
- Every `type` is from the taxonomy.
- Every `id` was generated by `generate_ids.py` (format: `<module>-v-<name>-<random>`).
- No proper nouns as standalone learnable items.

---

### Phase 2: Generate Grammar Concepts

- Generate **2–4** grammar concepts that cover the module's **grammar focus** and support its communication goal.
- Concepts **must** be drawn from the canonical taxonomy in idea.md §3.3, respecting the **"Available from" CEFR gate** — do not introduce a concept ahead of the module's level.
- For **each** concept, generate the `explanation` (short, in English, pitched at the module's CEFR level) and **1–2 `examples`**, each with `danish` + `english`. This is required — Step 1 of the module flow (idea.md §3.1.1) renders these.
- **Before writing the file**, run `generate_ids.py` for all grammar concepts in one call (see ID Rules). Use the returned ids verbatim.

**Output schema** (array of `GrammarConcept`, matching data-model.md):
```json
{
  "id": "A2-02-g-inversion-4821",
  "name": "Inversion (verb-second rule)",
  "category": "sentence_structure",
  "cefrLevelIntroduced": "A2",
  "explanation": "When a sentence starts with something other than the subject, the verb stays in second position and the subject follows it.",
  "examples": [
    { "danish": "I dag har jeg et møde.", "english": "Today I have a meeting." },
    { "danish": "Om sommeren rejser vi meget.", "english": "In the summer we travel a lot." }
  ]
}
```

**Expected file:** `<user-specified-folder>/<module>/<module>-grammar.json`.

**Self-validation before advancing:**
- 2–4 concepts, each present in the taxonomy and allowed at this CEFR level.
- Each concept has an `explanation` and 1–2 valid `examples`.
- Every `id` was generated by `generate_ids.py` (format: `<module>-g-<name>-<random>`).

---

### Phase 3: Generate Exercises

- Read the generated vocabulary in `<user-specified-folder>/<module>/<module>-vocabulary.json`.
- Read the generated grammar concepts in `<user-specified-folder>/<module>/<module>-grammar.json`.
- Generate the exercise bank. You **MUST** follow every rule in `rules-for-generation.md` (bundled with this skill) — this is **important**. Use `module-examples.md` as the style reference.
- Every `vocabularyItemId` and `grammarConceptId` in an exercise **must** reference an `id` that exists in the files from Phases 1–2. Never invent ids here.
- **Coverage requirement (MUST):** **every** vocabulary item must have **at least one** exercise — ideally **two** — and **every** grammar concept must have at least one (idea.md §3.1.3). This is a hard requirement, enforced by `validate_coverage.py` in Phase 4; the bank is not complete until the script exits 0. Bank size emerges from coverage, not a fixed target — aim for two exercises per vocabulary item from the start so Phase 4 passes cleanly.

**Expected file:** `<user-specified-folder>/<module>/<module>-exercises.json`.

**Self-validation before finishing:**
- Every exercise has `moduleId` set to the module's id (e.g. `danish-A2-02`). Never `null` — level-test exercises use a separate generation path not covered by this skill.
- Exactly one of `vocabularyItemId` / `grammarConceptId` set per exercise, per the type→link table in `rules-for-generation.md`.
- Every referenced id exists in the Phase 1/2 files.
- Every vocab item has ≥1 exercise (target: 2) and every grammar concept has ≥1 exercise. This is verified by `validate_coverage.py` in Phase 4 — do not rely on a manual count.
- `timesShown` is `0` everywhere.
- `translation_active`: `alternativeAnswers` covers word-order variants and paraphrases; canonical `answer` is the most natural spoken form.
- `sentence_reorder`: if multiple word orderings are valid, canonical `answer` is the most natural spoken form and `alternativeAnswers` lists the others; `words` is unchanged (same tiles).
- `error_correction`: if the corrected sentence admits multiple word orders, canonical `answer` is the most natural spoken form and `alternativeAnswers` lists the others.
- `fill_blank`, `multiple_choice`, `conjugation_drill`: `alternativeAnswers` is always `[]`.
- Every `sentence_reorder` exercise has a `words` array (shuffled tokens, no trailing punctuation) and its `prompt` is the English translation; `promptTranslation` is `null`.
- Bank composition falls within the CEFR-level distribution targets in `rules-for-generation.md`. If the coverage requirement forces a type over its ceiling, note it explicitly.

### Phase 4: Coverage & Distribution QA

Two hard gates run against the generated exercise bank, in order. **Do not mark the module complete until both scripts exit 0.** Coverage runs first — a bank that fails coverage must be fixed before distribution matters.

#### Step 1 — Coverage gate (MUST)

Verify that every vocabulary item has at least one exercise (target: two) and every grammar concept has at least one. This is the binding requirement from idea.md §3.1.3.

```bash
python3 .claude/skills/generate-module-content/validate_coverage.py \
  <module> <user-specified-folder>/<module>/<module>-exercises.json
```

The script auto-detects the sibling `<module>-vocabulary.json` and `<module>-grammar.json`.

**Reading the output:**

| Result | Meaning | Action |
|---|---|---|
| `PASS` (full coverage) | Every vocab item has ≥2 exercises; every grammar concept ≥1 | Done |
| `PASS` (with WARN) | Every vocab item has ≥1 exercise, but some have only 1 (below the target of 2) | Add a second exercise for the listed items where the type distribution allows; acceptable to proceed if distribution targets would otherwise break |
| `FAIL` | One or more vocab items or grammar concepts have **zero** exercises, or an exercise references a non-existent id | **MUST fix.** Generate the missing exercises (valid id references only), then rerun. Repeat until exit 0 |

**This loop is mandatory:** if the script exits 1, generate the missing exercises and rerun. Do not advance to Step 2, Phase 5, or declare the module complete while coverage FAILs. To additionally enforce the two-per-item target as a hard gate, run with `--strict` (optionally `--target N`).

#### Step 2 — Distribution QA

Once coverage passes, verify the exercise-type distribution against the CEFR-level targets.

```bash
python3 .claude/skills/generate-module-content/validate_distribution.py \
  <module> <cefrLevel> <user-specified-folder>/<module>/<module>-exercises.json
```

**Reading the output:**

| Result | Meaning | Action |
|---|---|---|
| `PASS` | All type percentages within targets | Done |
| `PASS with WARN` | MC over ceiling but fully explained by coverage-forced exercises | Done — note the warning in your summary to the user |
| `FAIL` | One or more types outside target range | Fix, then rerun |

**How to fix a FAIL:**

The script prints a "Required corrections" block with exact counts, e.g.:

```
✗ multiple_choice: 69.1% (target ≤ 45%) — replace ~14 with other types
✗ translation_active: 7.3% (target ≥ 10%) — add ~2 more exercises of this type
```

- For surplus types: select exercises of that type that test vocabulary items already covered by other types; replace them with the deficit type(s), keeping the same `vocabularyItemId` or `grammarConceptId`.
- For deficit types: generate new exercises of that type for vocabulary items or grammar concepts that are under-represented.
- Do not add exercises without a valid id reference — every `vocabularyItemId` and `grammarConceptId` must still exist in the Phase 1/2 files.
- After editing the exercises file, rerun the script. Repeat until exit 0.

**Surface to the user:** one line per gate — e.g. `Coverage: PASS (24/24 vocab items, 18 at target 2)` and `Distribution QA: PASS (55 exercises)`, or `Coverage: FAIL (2 uncovered) → corrected → PASS`.

### Phase 5: Module generation

Generate the `Module` record that ties all three artifacts together.

**Step 1 — Extract IDs from the generated files** by running `extract_module_ids.py`:

```bash
python3 .claude/skills/generate-module-content/extract_module_ids.py \
  <module> <user-specified-folder>
```

Example:
```bash
python3 .claude/skills/generate-module-content/extract_module_ids.py \
  A2-02 content
```

The script reads `<user-specified-folder>/A2-02/A2-02-vocabulary.json` and `…/A2-02-grammar.json` and outputs:
```json
{
  "vocabularyItemIds": ["A2-02-v-kaffe-7392", "..."],
  "grammarConceptIds": ["A2-02-g-inversion-4821", "..."]
}
```

If the script exits non-zero, stop and surface the error to the user — do not proceed with a hand-collected list.

**Step 2 — Write the module file** using the script's output verbatim for `vocabularyItemIds` and `grammarConceptIds`:

- **`id`**: `danish-<cefrLevel>-<sequence>` (see ID Rules → Module ID). E.g. `danish-A2-02`.
- **`title`**: The module title from `default-modules.md` (e.g. `"A Day in the Past (Simple Past)"`).
- **`theme`** and **`communicationGoal`**: Taken verbatim from the module shell in `default-modules.md`.
- **`cefrLevel`**: The level from `default-modules.md` (e.g. `"A2"`).
- **`vocabularyItemIds`**: Use the array from the script output — do not hand-write or re-derive.
- **`grammarConceptIds`**: Use the array from the script output — do not hand-write or re-derive.
- **`createdAt`**: ISO 8601 timestamp of generation (use the current date/time).
- **`isUserGenerated`**: `false` for all default modules.

**Output schema** (matching `data-model.md`):
```json
{
  "id": "danish-A2-02",
  "title": "A Day in the Past (Simple Past)",
  "theme": "Narrating past events",
  "communicationGoal": "Describe what happened yesterday, last week, or in a past situation",
  "cefrLevel": "A2",
  "vocabularyItemIds": ["A2-02-v-kaffe-7392", "..."],
  "grammarConceptIds": ["A2-02-g-inversion-4821", "..."],
  "createdAt": "2026-05-31T00:00:00Z",
  "isUserGenerated": false
}
```

**Expected file:** `<user-specified-folder>/<module>/<module>.json` (e.g. `A2-02/A2-02.json`).

**Self-validation before finishing:**
- `extract_module_ids.py` exited 0.
- `vocabularyItemIds` and `grammarConceptIds` are copied verbatim from the script output — no manual edits.
- `id` matches the `danish-<cefrLevel>-<sequence>` pattern and corresponds exactly to the target module.
- `isUserGenerated` is `false`.
- Every `moduleId` in the Phase 3 exercise file equals the `id` in this file.

Surface a one-line summary to the user: e.g. `Module: danish-A2-02 written (23 vocab items, 3 grammar concepts, 55 exercises)`.

## Red Flags

- Starting anything without having asked the user the folder path for generated files.
- Starting anything without knowing which module to generate content for.
- Writing any code other than running `generate_ids.py` to mint ids.
- Generating exercises before vocabulary items and grammar concepts exist.
- Generating exercises whose `vocabularyItemId` / `grammarConceptId` do not exist in the Phase 1/2 files.
- Declaring the module complete while `validate_coverage.py` (Phase 4, Step 1) still FAILs — leaving any vocabulary item or grammar concept with zero exercises is a hard violation.
- Declaring the module complete without running `validate_distribution.py` (Phase 4, Step 2).
- Writing `vocabularyItemIds` or `grammarConceptIds` by hand in Phase 5 instead of using `extract_module_ids.py`.
- Inventing JSON fields not present in `data-model.md`.
- Producing grammar concepts without an `explanation` and `examples`.
- Introducing a grammar concept ahead of the module's CEFR level.
